# 项目概览

## 项目目标

- 构建一个《杀戮尖塔2》AI 建议型 mod，在游戏内针对路线、选卡、遗物给出推荐，而不是替玩家执行操作。
- 采用可迭代架构：阶段一由游戏侧 mod 内置本地规则评分器完成路线推荐 MVP；后续阶段再引入 Python AI 服务承接更复杂的候选生成、评分与解释。
- 优先交付一个可运行、可观察、可扩展的 MVP，再逐步扩展到更多决策点。

## MVP 范围

MVP 仅覆盖“路线推荐”能力，目标是先打通一条最短可用链路。

- 在地图决策场景中读取当前运行状态。
- 将地图、当前位置、可达节点和玩家关键状态标准化为统一数据结构。
- 由本地 baseline 规则评分器生成路线候选评分结果。
- 在游戏内展示 Top-K 路线推荐、分数和简短理由。
- 记录推荐结果和玩家实际选择，作为后续调参与离线评估基础。

## 非目标

- 不做自动战斗。
- 不做点击模拟、输入注入或任何形式的自动代打。
- 不做整局代理，不接管全流程策略执行。
- 不在首个里程碑中引入复杂训练流水线、联网依赖或大规模模型基础设施。
- 不在尚未打通路线推荐前，同时铺开选卡、遗物、战斗内行为等全部能力。

## 阶段顺序

1. 项目初始化：文档、模块边界、目录骨架、协议草案。
2. 路线推荐 MVP：地图状态读取、路线候选生成、baseline 评分、游戏内推荐 UI。
3. 观测闭环：决策日志、重放样本、基础离线评估。
4. 选卡推荐：奖励牌状态建模、候选评分接口、结果展示。
5. 遗物推荐：遗物价值建模、候选评分接口、结果展示。
6. 策略迭代：基于日志和离线评估持续优化规则、特征和解释质量。

## 成功标准

- 玩家在不改变手动操作前提下，能在游戏内看到稳定、可读、可解释的路线推荐。
- 路线推荐链路完整可用：状态采集、JSON 序列化、评分请求、结果展示全部打通。
- Python baseline scorer 能返回可排序的 Top-K 路线结果，并附带简短理由。
- 请求/响应与玩家选择被记录下来，可用于后续分析和调优。
- 项目结构支持后续扩展到日志系统、Python 服务、选卡与遗物推荐，而无需重写核心数据结构和展示方式。
